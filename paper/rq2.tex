To understand how these developers work, we need to understand \emph{which} agent harnesses they use and how they use them.
We hypothesize that hyperactive developers use different coding agent harnesses, such as Claude Code, OpenAI Codex, or Pi, depending on the task, and may change their primary harness over time.

\todo{add an identifier per hypothesis, such that you can refer to them in the result secton}Our hypothesis is that agent usage increases over time both in frequency and volume, that developers sometimes switch their primary harness, that Claude Code is the most popular harness, that different harnesses are used for different tasks, and that the same harnesses are used across repositories.
\todo{this is a result, it's impossible to have this in the methodology section}We note that the harnesses themselves evolved rapidly during our observation window (e.g., Cursor expanded from completions to agentic workflows), so observed changes may reflect tool availability as much as developer behavior.
To test this, we analyze each commit's metadata and changed files for signals that indicate the use of a specific harness. We assume that all code is generated with some harness, but that not every commit carries a detectable signal. We do note that there are instances of what appear to be fully-manual commits\todo{what's the oracle for that?} but those are entirely drowned out by generated ones, so assuming the use of a harness for all commits is a reasonable approximation of the actual behavior.

We assign a harness to a commit if the commit mentions a harness in its author list or co-author trailers, or if the commit changes agent-specific configuration files.
Notably, attribution signals are uneven: harnesses like Claude Code and Codex append co-author trailers by default,\todo{do we have a primary source to backup this claimi wold be great (but I agree this is almost "common knowledge"} while others (e.g., Copilot, Cursor) leave no commit-level trace, so the absence of a signal does not imply the absence of a harness.

To gain even more insight into what they use agents for, we also analyze the type of work done in the commit based on the categories in the Conventional Commits specification\footnote{\url{https://www.conventionalcommits.org/en/v1.0.0/}}.
Robbes et al.~\cite{Robbes2026} found that agent-assisted commits tend to adhere to these conventions well, which allows us to use them as a reliable categorization.
This reveals which tasks developers use agents for and how they distribute different harnesses across their workflow.

Since our goal here is to study what harnesses are used for, we require a harness assignment for every commit. When no signal is found, we assign the commit to a repository- or developer-specific fallback harness. To determine per-repository default harnesses, we extend the analysis to additional areas of interest, such as pull requests, code reviews, and issue triage, all of which can also carry agent signals.
